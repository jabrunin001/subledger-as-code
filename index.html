The problem

The control that ties out can still be wrong

Every accountant's first check is that debits equal credits. Necessary, but not enough. Misroute a payment's interest into the wrong account and the books still balance to the penny: the entry is internally consistent, the trial balance ties out, and the money is in the wrong place.

Catching that needs a substantive control: re-derive what each balance should be straight from the source events, then reconcile it against what the ledger actually posted. That's the control auditors lean on at close, and it's the one this project implements in code, in data, and in CI.

The system

A posting engine, then two invariants

Every event becomes balanced double-entry legs through a data-driven posting engine. The rules live in a seed file, not in hardcoded SQL. Two invariants hold the whole thing together: every entry's debits equal its credits, and the ledger balances match balances re-derived independently from the source events.

The default target is DuckDB, so the repo clones and runs with no warehouse, no credentials, and no network. A Snowflake target adds RBAC, masking policies, and clustering for the same models.

From detection to triage

The evidence pack explains the exception

Catching the break is v1. Explaining it is v2. A deterministic triage step groups the variances, works out the root cause, points at the responsible posting rule, and writes it up in plain English, with no LLM involved. The pack command embeds this in a checksummed, tamper-evident evidence pack, and CI asserts the classification.

▣ Deterministic triage · classification: wrong_account · high confidence

cash + interest_income: value reallocated, not lost

Accounts cash and interest_income diverge from their source-derived expectations by offsetting amounts (net +0.00 ≈ 0: cash +4,394.83, interest_income −4,394.83), indicating value was reallocated between them rather than created or lost. That points to a posting leg targeting the wrong account.

Both accounts are posted by the payment entry; the leg crediting interest_income (rule payment/3, amount_source interest) is the leading suspect for misrouting.

Next step: inspect the candidate posting rules for a leg posting to the wrong account, correct it, and re-run reconciliation. An optional local Ollama backend rewrites this prose more fully, and it never changes the deterministic root cause.
